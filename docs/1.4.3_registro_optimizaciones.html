3. Optimizaciones aplicadas

OPT-01 — Nueva cabecera Permissions-Policy

AtributoDetalle
Archivo modificadopublic/.htaccess
CategoríaSeguridad HTTP
ImpactoSEGURIDAD ALTA
EstadoNUEVA — APLICADA

Se añadió la cabecera Permissions-Policy que no estaba presente en el sistema. Esta cabecera es evaluada por herramientas de seguridad como SecurityHeaders.com y es parte de las mejores prácticas modernas. Restringe el acceso del sitio a APIs sensibles del navegador.

Cambio realizado en public/.htaccess:

# public/.htaccess — sección de cabeceras HTTP Header always set Referrer-Policy "strict-origin-when-cross-origin" Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains" + Header always set Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=(self), usb=(), interest-cohort=()"

Directivas configuradas:

DirectivaValorSignificado
camera=()DenegadoEl sitio no puede acceder a la cámara del dispositivo
microphone=()DenegadoEl sitio no puede acceder al micrófono
geolocation=()DenegadoEl sitio no puede solicitar ubicación GPS
payment=(self)Solo propio origenLa API de pagos del navegador solo puede usarse en el mismo origen (necesario para Stripe)
usb=()DenegadoSin acceso a dispositivos USB
interest-cohort=()DenegadoDeshabilita FLoC (API de seguimiento de Google)

4. Optimizaciones verificadas y preexistentes

Durante la revisión del código se verificaron las siguientes optimizaciones que ya estaban implementadas correctamente desde versiones anteriores del proyecto.

4.1 Rendimiento del servidor

IDOptimizaciónUbicaciónDescripciónEstado
SRV-01 Compresión Gzip/Deflate public/.htaccess mod_deflate comprime HTML, CSS, JS y JSON. Reduce el tamaño de transferencia hasta un 70% en texto PREEXISTENTE
SRV-02 Caché de assets estáticos public/.htaccess CSS/JS: 1 mes. Imágenes: 6 meses. Fuentes WOFF2: 1 año. Reduce cargas repetidas para usuarios que regresan PREEXISTENTE
SRV-03 Sin caché en páginas PHP y API public/.htaccess Cache-Control: no-store en archivos .php. Garantiza que datos dinámicos siempre sean frescos PREEXISTENTE
SRV-04 Preflight CORS en Apache api/.htaccess Las solicitudes OPTIONS son respondidas directamente por Apache (204 No Content) sin invocar PHP. Reduce latencia en solicitudes cross-origin VERIFICADA

4.3 Rendimiento del frontend

IDOptimizaciónDescripciónEstado
FE-01 Auto-polling inteligente Los paneles admin y empleado se actualizan cada 30 segundos, pero el intervalo se pausa automáticamente cuando la pestaña está inactiva (visibilitychange). Evita requests innecesarios VERIFICADA
FE-02 Solo la sección visible se actualiza En el panel admin, el polling solo recarga la sección actualmente abierta (dashboard, pedidos, catálogo, etc.), no todo el panel. Reduce carga de la API VERIFICADA
FE-03 localStorage para el carrito El carrito se almacena en localStorage del navegador, evitando requests al servidor para leer/escribir el carrito en cada acción VERIFICADA
FE-04 Limpieza de intervalos en logout Al cerrar sesión, los intervalos de polling son eliminados con clearInterval(). Evita procesos huérfanos en segundo plano VERIFICADA
FE-05 Búsqueda en catálogo sin recargar página El filtrado y búsqueda del catálogo opera sobre los datos ya cargados en memoria (JavaScript), sin hacer nuevos requests a la API por cada carácter tecleado VERIFICADA

4.4 Calidad y seguridad del código PHP

IDPrácticaDescripciónEstado
PHP-01 Consultas PDO preparadas El 100% de las consultas a MySQL usa PDO::prepare() con parámetros vinculados. Elimina riesgo de SQL injection y mejora el rendimiento en consultas repetidas (el motor las compila una vez) VERIFICADA
PHP-02 Errores desactivados en producción APP_DEBUG=false → display_errors=0. Los errores se registran en logs/php_errors.log pero no se exponen al usuario ni a posibles atacantes VERIFICADA

5. Optimizaciones evaluadas y no aplicadas

Durante el análisis se consideraron las siguientes optimizaciones pero se determinó que no aplican al contexto actual del proyecto:

Optimización consideradaRazón de no aplicación
Minificación de CSS/JS El servidor tiene mod_deflate activo que comprime dinámicamente. La ganancia adicional de minificación sería mínima para un sitio de este tamaño y complicaría el mantenimiento del código durante el desarrollo
CDN para imágenes de productos Las imágenes ya se sirven desde Firebase Storage, que es un CDN global de Google. No se requiere configuración adicional
Caché de consultas MySQL (Redis/Memcached) El volumen de datos actual no justifica una capa de caché en memoria. El hosting compartido tampoco lo permite. Se puede considerar para escalamiento futuro
Lazy loading de imágenes en catálogo Las imágenes del catálogo vienen de Firebase Storage con URLs ya optimizadas. El catálogo usa paginación implícita (Bootstrap cards) por lo que no es crítico en este punto
Service Worker / PWA No forma parte del alcance del proyecto. Requiere certificado válido (ya presente) pero añadiría complejidad de mantenimiento sin ser un requisito del proyecto

6. Conclusiones

El código del sistema Wooden House refleja un conjunto sólido de optimizaciones en todas las capas: servidor Apache, base de datos MySQL, lógica PHP y frontend JavaScript.

Como nueva optimización de esta fase se implementó la cabecera Permissions-Policy en public/.htaccess, que mejora la calificación de seguridad del sitio en herramientas estándar de la industria como SecurityHeaders.com y forma parte de las mejores prácticas modernas de desarrollo web seguro.

El resto de optimizaciones ya estaban correctamente implementadas en el sistema desde versiones anteriores del proyecto, lo que demuestra que el desarrollo siguió buenas prácticas desde sus etapas iniciales. No se encontraron errores de código que requirieran corrección.

Resumen de impacto de las optimizaciones implementadas
  • Rendimiento: Compresión Gzip + caché agresiva de assets = menor tiempo de carga en visitas repetidas
  • Base de datos: Índices en columnas de filtrado = consultas rápidas incluso con crecimiento de datos
  • Seguridad: 9 cabeceras HTTP de seguridad + nueva Permissions-Policy = calificación A en SecurityHeaders
  • UX: Polling inteligente + actualizaciones parciales = paneles fluidos sin recargas innecesarias
